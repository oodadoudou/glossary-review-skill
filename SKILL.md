# Glossary Review — 小说术语表审查 Skill

## 何时触发

用户输入 `/glossary-review [目录路径] [--lang ko|ja|auto]` 时。

参数：
- `目录路径`：**必需**。必须等待用户显式提供目录路径（通过参数、`@` 引用、或后续消息）后才能开始审查。**绝对不要根据工作目录或上次会话自动猜测目录并开始审查。**
- `--lang`：可选，`ko`（默认）/ `ja` / `auto`（按术语原文自动判定）。

示例：
- `/glossary-review ./novels/some_novel` — 指定目录，韩语
- `/glossary-review ./novels/some_novel --lang ja` — 指定目录，日语
- `/glossary-review --lang auto` — 等待用户提供目录后自动检测

**若用户未提供目录路径**：询问用户 "请提供要审查的术语表目录路径"，等待回复后再继续。不得自行扫描或猜测。

## 工作流（Phase 1–7）

### Phase 1 — 文件发现与初始化

**扫描目标目录，按优先级定位 xlsx**：
1. 文件名包含 `final`（不区分大小写）
2. 文件名包含 `output`
3. 目录中第一个 `.xlsx`（排除 `~` 临时锁文件 与 `modified*`）

**必需文件**：
- `output_detail.txt` — 参考文本。缺失则报错并停止。
- `background.txt` — 小说背景。**缺失则向用户提问**："未检测到 background.txt，请用一段话描述小说背景（题材、世界观、主要角色与关系）。"用户回复后，将其写入 `background.txt`，再继续。

**xlsx 列结构识别**（不强制列名，按位置 + 关键词容错）：
- 第 1 列 → `src`（原文）
- 第 2 列 → `dst`（译文）
- 名为 `info` / `备注` 的列 → 分类
- 名为 `regex` 的列 → 正则（保留不动）
- 名为 `frequency` / `次数` / `freq` / `count` 的列 → 词频
- 其他所有列 → 保留原列名与 dtype 不动

### Phase 2 — 语言判定

**若 `--lang auto` 或未指定**：
- 抽取 `src` 列前 50 个非空值
- 含 `[\uAC00-\uD7AF]` (한글) 占比 > 30% → 判定 `ko`
- 含 `[\u3040-\u309F\u30A0-\u30FF]` (ひらがな + カタカナ) 占比 > 20% → 判定 `ja`
- 否则默认 `ko`，并在对话中告知用户："自动检测置信度低，已使用韩语规则；如需日语请加 `--lang ja`。"

**若用户显式 `--lang ko/ja`**：直接采用，不检测。

### Phase 3 — 加载与分组

**加载 xlsx**（关键：保留 dtype）：
- 使用 `pd.read_excel(..., engine='openpyxl')`，**不要** 强制 `astype(str)`
- 数值列保持 `int64` / `float64`；空单元格保持 `NaN` 而非 `''`
- 记录原始列顺序与 dtype 字典，写回时按原 dtype 还原

**解析 `output_detail.txt`**：
- 按 `原文：` 分块，提取每块的术语 → 参考文本映射
- 若文件中无 `原文：` 标记（纯小说原文），则对每个术语在文中做关键词扫描，取首次出现的前后 1 行作为上下文

**分组与排序**（术语处理顺序）：

按以下大类顺序处理，**组内按 frequency 降序**：
1. **角色**（备注含：角色 / 男性角色 / 女性角色 / 动物与非人角色 / 历史与知名人物 / 群体代称 / 称呼与头衔 / ID与外号）
2. **地点**（地理 / 地名 / 命名地理）
3. **组织**（命名组织 / 组织机构 / 派系）
4. **设定**（小说设定 / 核心背景设定 / 网文机制）
5. **物品**（物品 / 道具）
6. **能力技能**
7. **其他/未分类**

未分类的术语放最后，按 frequency 降序处理。

### Phase 4 — 逐词审查（核心，无批处理）

**对每个术语，独立执行**：

1. **构造审查上下文**：
   - 原文 / 当前译文 / 当前分类 / 出现次数
   - 该术语在 `output_detail.txt` 中的参考文本
   - 小说 background.txt 全文
   - 已审查过的同组术语的判定结果（用于即时一致性比对）

2. **网络搜索**（按需，**通用词跳过**，特殊术语必搜）：

   **跳过搜索（不需要）**：
   - 通用国家/城市/地名（런던/뉴욕/한국 等已有标准译法）
   - 通用职业/称谓（비서/교수/대표 等）
   - 通用日常词（无搜索价值，直接按规则判定）
   - 已与 background.txt 一致的主角名（Tier S 已锁定）

   **必须搜索（特殊术语）**：
   - **作品自创/虚构设定名称**（如本作的 `네임` / `에보니 캐슬` / `아젠포드`）— 验证是否为已知 IP 或作者原创
   - **疑似引用的真实历史人物/作品**（如 `타오위안밍` = 陶渊明）— 确认身份后正确分类
   - **可能涉及版权 IP 的衍生术语**（玩具/动漫/游戏角色名等，如 `츄니핑`/`하트츄`）— 查官方中文译名
   - **冷门外来语/网络流行词** — 来源网站验证（DC Inside / 더쿠 / 5ch / Twitter / pixiv）
   - **判定有疑问的人名转写**（如音译可能错配 `베이른` ≠ `바이런`）— 验证转写规则
   - **重要长文本/品牌/机构虚构名**（涉及作品世界观）

   **搜索关键词组合**：
   - `"<原文>" + 현대문학/웹소설/小说` 或 `"<原文>" + 维基/나무위키/pixiv百科/なろう wiki`
   - 用 `WebSearch` 工具，必要时再 `WebFetch` 进具体页面验证
   - 搜索结果用于：① 验证翻译准确性 ② 识别是否为已知 IP/作品 ③ 判断是否为网文流行表达

3. **Tier 判定**：
   - 出现于 `background.txt` → **Tier S**（核心设定，绝对禁删）
   - frequency ≥ 5 → **Tier A**（高频，重点排查通用词错提）
   - frequency = 4 → **Tier B**（普通）
   - frequency ≤ 3 → **Tier C**（低频，大胆建议删除通用词）

4. **应用审查规则**（详见下方 §共用规则集 + §语言专属规则块）。

5. **产出单条结果 JSON**：
```json
{
  "src": "原文",
  "current_dst": "当前译文",
  "recommended_dst": "建议译文",
  "should_delete": false,
  "current_category": "原分类",
  "suggested_category": "大类/子类",
  "tier": "A",
  "search_summary": "网络搜索关键发现（一句话）",
  "justification": "判定理由（含上下文/搜索/规则引用）",
  "action": "Keep | Modify | Delete | Recategorize"
}
```

### Phase 5 — 全局一致性扫盘

所有术语初判完成后，**回头横扫**：

1. **角色组聚类**：同一角色的全名/省略名/昵称视为术语组
   - 例（韩）：`이해든 → 李海灯` / `해든 → 海灯`
   - 例（日）：`佐藤太郎 → 佐藤太郎` / `太郎 → 太郎` / `佐藤さん → 佐藤先生`
   - 强制组内**性别一致**：任一形式被判男性/女性，全组继承
   - 强制中文译名**同源**：全名包含省略名

2. **组织组聚类**：全称 ↔ 简称 ↔ 内部俗称 → 中文译名同源
   - 例：`MCE 뮤직 → MCE音乐` / `MCE → MCE`

3. **两字姓名保护**：即使存在全名条目，两字名条目**不删除**

4. **修正不一致项**：覆盖 Phase 4 的初判，并在 `justification` 中标注"经一致性扫盘修正"

### Phase 6 — 提案输出（**严格顺序：先写文档 → 后回对话简表**）

**步骤顺序不可颠倒**：
1. 先 **6a 写入 `review_proposal.md`**（完整逐条审查留痕）
2. 然后 **6b 在对话中回复简化表格**（仅含变更项，引用文档路径）

文档先于对话表是因为：用户应能在 IDE/编辑器中查看完整审查记录后，再决定如何回复对话简表。

---

**6a. 写入 `review_proposal.md` — 必须逐条记录每一个术语的判定**（不仅是有变更的）：

**核心要求（强制）**：
- 文档必须包含**全部 N 条术语**的逐条判定记录，不允许省略或节选
- 每条必须明确标注 action: `Keep` / `Modify` / `Delete` / `Recategorize`
- 即便是 `Keep`（无变更），也需写出**判定理由**（哪怕一句话）和必要的 Tier/分组确认
- 对话内 6a 简表只列变更项以减少噪音；但 6b 文档是**审查留痕的源文件**，用户应能在此核查任意一条术语的处理逻辑

**结构**：

```markdown
# Glossary Review Proposal
- 目录: <path>
- 语言: ko/ja
- 共 N 条术语 — Modify M / Delete D / Recategorize R / Keep K

---

## 变更项（Modify / Delete / Recategorize）

### #2 가웬 저택 (Modify)
- 当前译文: 加温公馆
- 建议译文: 加文公馆
- 当前分类: 命名地理 → 建议: 地点/命名地理
- Tier: C (frequency=2)
- 网络搜索: 韩文小说中"가웬"为虚构家族名，无固定译法...
- 上下文引用: "겹벚꽃이 흐드러지게 핀 4월이었다..."
- 判定理由: 与同组术语 `가웬 → 加文`、`가웬 가문 → 加文家族` 译名同源...

---

## 保持项（Keep）— 必须逐条列出

按分组顺序（角色 → 地点 → 组织 → 设定 → 物品 → 能力 → 其他），组内按 freq 降序：

### 角色组（X 条）
- `이해든` → 李海灯 (Tier S, freq=N) — Keep。Background 一致，与短名 `해든 → 海灯` 同源 ✓
- `해든` → 海灯 (Tier A, freq=N) — Keep。短名形式，性别男性已锁定。
- `에드윈` → 埃德温 (Tier S, freq=N) — Keep。Background 一致 ✓
- ...（每一条都列出）

### 地点组（X 条）
- `젠도프` → 詹多夫 (Tier C, freq=13) — Keep。虚构地名音译合理，无歧义。
- ...（每一条）

（其余分组同样）
```

**最低记录粒度**（针对 Keep 项）:
- 必须含：原文 / 译文 / Tier / freq / 一句话理由
- 网络搜索结果可省略（仅当变更项需要）
- 上下文引用可省略（仅当变更项需要）

不得：
- ❌ "已审查保持不变（节选 N 条）" 这种省略写法
- ❌ "其余按上下文判定无问题" 这种打包归类
- ❌ "节选关键术语" 而留其他不审

**6b. 对话内简化 Markdown 表**（**强制要求**：写完 `review_proposal.md` 之后，必须立即在对话中输出此表）：

仅列出 **action ≠ Keep** 的条目（Modify / Delete / Recategorize），按动作类型分组：

**强制要求**：
- 文档写入完成后，**必须立即输出此简表**，不得省略
- 即便没有变更（M=D=R=0），也要输出 "审查完成 — 共 N 条术语全部 Keep，无建议变更" 的明确陈述
- 对话简表只出现 **Modify / Delete / Recategorize**，**不得**列 Keep 条目（避免噪音；Keep 详情已在 `review_proposal.md`）

### Phase 7 — 用户反馈循环

- 用户回复 **"全部同意"** / **"yes"** → 进入 Phase 8 写回
- 用户对某条质疑（如 `"#2 为什么改？"` 或 `"我不同意 #5，应该保留"`）→ 给出**详细解释**：引用参考文本原句、网络搜索结果、规则条款；若用户坚持，**以用户意见为准**，更新提案后再次确认
- 用户提出新约束（如"所有称谓后缀全保留"）→ 重跑相关条目
- 持续循环直至用户最终确认

### Phase 8 — 写回 glossary_output_final.xlsx

**直接覆盖** 原 final 文件（不备份，用户有源 output 可追溯）：

- **保留所有原列**（src / dst / info / regex / frequency / 任何额外列）
- **保留原 dtype**：frequency 仍为整数，不被字符串化；NaN 保持 NaN
- **保留 autofilter**：写入时启用第一行自动筛选
- **被 Delete 的整行删除**
- **Modify 的行**：仅更新 `dst` 与 `info`（若分类变更），其他列原样
- 写入后输出："已写回 `<path>/glossary_output_final.xlsx`，共 N 条变更已应用。"

**实现伪代码**（参考）：
```python
import pandas as pd

# 读取（保留 dtype）
df = pd.read_excel(path, engine='openpyxl')
original_cols = df.columns.tolist()
original_dtypes = df.dtypes.to_dict()

# 应用变更：删除 + 修改
df = df[~df['src'].isin(deleted_terms)]
for term, new_dst, new_cat in modified:
    mask = df['src'] == term
    df.loc[mask, 'dst'] = new_dst
    if 'info' in df.columns and new_cat:
        df.loc[mask, 'info'] = new_cat

# 还原 dtype
for col, dt in original_dtypes.items():
    if col in df.columns:
        try:
            df[col] = df[col].astype(dt)
        except (ValueError, TypeError):
            pass  # 跳过无法回还的列（如含 NaN 的 int → float）

# 写入（autofilter，xlsxwriter）
with pd.ExcelWriter(path, engine='xlsxwriter') as writer:
    df.to_excel(writer, index=False, sheet_name='Sheet1')
    ws = writer.sheets['Sheet1']
    rows, cols = df.shape
    if rows > 0:
        ws.autofilter(0, 0, rows, cols - 1)
```

## §共用规则集（韩 / 日 通用）

### 1. 人名一致性
- 已确立人名映射视为**强绑定**，全名形式 ↔ 省略形式必须中文同源
- 同一角色不同名称形式视为"同一人名术语组"，组内**性别一致、身份一致、立场一致**
- 一旦任一形式被判定为男性/女性，全组继承
- **两字姓名保护**：两字人名条目即使存在全名条目也不得删除

### 2. 组织一致性
- 组织、机构、团体、势力、公司、学校、帮派均为专有名词
- 全称 ↔ 简称 ↔ 俗称的中译必须同源
- 已确立译名不得因风格偏好擅自更换

### 3. 精简原则（核心）

**判定核心问题**：「这个术语**是否被算法误提取为术语**？是否其实只是日常词汇 / 多义需上下文判定的词？」
- ✅ 是 → **建议删除**（不该出现在术语表中）
- ❌ 否 → **保留**（即便译法标准/通用，仍属术语保护范畴）

**关键认知更正**：术语表的目的是**确保专有名词稳定**与**给译者选词指引**。
- "译法标准化无歧义" 本身**不是删除理由**（런던 → 伦敦 译法标准，但仍属专有地名，应保留以确保不出错）
- 真正应删除的是**不该被算法当作术语**的通用日常词

#### 删除候选（6 类，不论 Tier 高低）

1. **通用日常物品/家具/衣物/身体部位/食品**（被误提取，无故事术语身份）
   - 例：`침대 시트` → 床单 — 通用家居物品 → 删除
   - 例：`옷` → 衣服 / `신발` → 鞋 / `컵` → 杯子 / `의자` → 椅子 → 删除
   - 例：`손` → 手 / `머리` → 头 / `눈` → 眼睛 → 删除
   - 例：`빵` → 面包 / `우유` → 牛奶 → 删除

2. **多义词需上下文判定**（强制单一译反而误译）
   - 例：`자리` → 座位/位置/职位（多义，需上下文判定具体含义）→ 删除
   - 例：`车座` 韩文（차좌/座席）→ 多义 → 删除
   - 例：`길` → 路/道路/方法（多义）→ 删除

3. **通用动词/形容词/副词被误提取为术语**
   - 例：`만남` → 相遇 / `사랑하다` → 爱 / 形态变化的动词
   - 例：日语 `ドキドキ` / `ふわふわ` 等拟声拟态语（Tier C 时大胆删）

4. **单字、纯字母数字、无歧义日常词** — 同上

5. **「XX的 XX」型修饰短语**（一律删除，不论频次）
   - 即「定语 + 中心词」组合，且中心词本身已是独立术语 / 或可直接由译者按上下文翻译
   - 例（韩）：`강무현의 차` → 姜武铉的车 / `태성제의 사무실` → 泰成济的办公室 / `우지민의 집` → 禹智珉的家 → 删除
   - 例（韩）：`서승원의 가족` → 徐承源的家人 / `김 대표의 비서` → 金代表的秘书 → 删除
   - 例（日）：`太郎の部屋` → 太郎的房间 / `主人公の友達` → 主人公的朋友 → 删除
   - **理由**：这种短语强制译为单一形式反而剥夺译者的语境判断（如 "～的家" 在不同语境可译为"家/家里/家中/府上"）；中心词本身若是术语应单独保留，搭配关系不需固化。
   - **例外**：「XX的 XX」整体已成为**有专属含义的固定专有名词**（如 `웃슨의 왕자님` → 沃森的王子 = 学校外号；`레빈가의 저택` = 特定建筑名）— 保留。

6. **低频且无规范价值的术语**（Tier C 且不属强制保护类）
   - 判定核心：「该术语是否真正对小说翻译起到**规范作用**？是否反而因强制单一译法**剥夺译者的语境判断**让翻译变得生硬？」
   - 删除条件（同时满足）：
     - frequency ≤ 3（Tier C）
     - 不属于强制保护类（非主角/核心地名/核心组织/核心设定）
     - 译法已通用、无歧义，译者按上下文自然翻译即可
     - 强制写入术语表反而限制译文灵活度
   - 例：`플라자 호텔` → 广场酒店（freq=1，纽约广场酒店，通用译法，无规范必要）→ 删除
   - 例：`삼성역` → 三成站（freq=1，首尔地铁站，通用地名）→ 删除
   - 例：`광화문` → 光化门（freq=1，韩国通用地名）→ 删除（除非剧情核心场景）
   - **认知更正**：之前「专有地名一律保留」过于宽松。Tier C + 通用无歧义 + 非剧情核心 = 删除，避免术语表臃肿。

#### 强制保护（即使通用也不删）

1. **Tier S**：出现于 background.txt 的核心设定 — 绝对禁删
2. **角色名**：所有人物（含历史人物引用、宠物、群体）
3. **专有地名**：即使是通行标准译法（런던/뉴욕/이탈리아 等）— 仍属术语，保留以避免错写
4. **职业/称号**：비서/교수/대표 等 — 即使通用也属"称号"类
5. **品牌名**：包括有官方中文的（테스코/스타인웨이 等）
6. **作品自创/虚构专有名词**：에보니 캐슬 / 아젠포드 / 네임 等
7. **同人多译名风险术语**：可能引发不一致的术语
8. **两字人名**：即使有全名条目也不删
9. **特定机构/餐厅/产品的衍生名**：场景/作品独特含义

#### 删除判定流程

```
术语 X
 ├─ 是 Tier S / 主角 / 高频(≥5)角色名 / 虚构专有名词? → 保留
 ├─ 是「XX的 XX」型修饰短语（且非固定专名）? → 删除
 ├─ 是通用日常物品/家具/衣物/身体部位? → 删除
 ├─ 是多义词需上下文判定? → 删除
 ├─ 是被误提的动词形态/拟声拟态? → 删除
 ├─ 是低频(≤3) + 通用无歧义 + 非剧情核心? → 删除
 ├─ 是核心剧情场景 / 主角组织 / 关键品牌? → 保留
 └─ 其他 → 综合判断：能否真正规范翻译？不能→删除；能→保留
```

#### 范例对齐

**删除（✅）**：
- `침대 시트 → 床单` — 删除（日常家居物品，无术语身份）
- `자리 → 座位/职位` — 删除（多义，需上下文）
- `만남 → 相遇` — 删除（动词性名词被误提）
- `강무현의 차 → 姜武铉的车` — 删除（XX的XX修饰短语，剥夺译者语境判断）
- `태성제의 사무실 → 泰成济的办公室` — 删除（同上）
- `플라자 호텔 → 广场酒店` (freq=1) — 删除（Tier C 通用地名，无规范价值）
- `삼성역 → 三成站` (freq=1) — 删除（Tier C 通用地铁站名）
- `필로소피 → Philosophy` — 删除并要求用户确认（违规：非英语原文不可直译英语；同时 Tier C 价值低）
- `퍼시픽디벨롭 → Pacific Develop` — 同上违规

**保留（❌ 不删）**：
- `에보니 캐슬 → 埃博尼城堡` — 保留（作品自创虚构地名）
- `아젠포드 → 阿真福德` — 保留（虚构地名，作品独有）
- `웃슨의 왕자님 → 沃森的王子` — 保留（XX的XX 但已成固定外号）
- `테스코 → 乐购` — 保留（品牌名，多角色重复出现）
- `JN → JN` — 保留（src 本身英文缩写，dst 自然保留）
- 高频角色名 / 主角组织 / 剧情核心场景 — 保留

#### 关键认知

- **标准化译法 + Tier C + 非剧情核心 = 删除**（避免术语表臃肿，剥夺译者语境判断）
- **高频或剧情核心**专有地名/品牌仍应保留（如全书反复出现的特定场所）
- 删除依据进化为：**「该术语是否真正对小说翻译起到规范作用？若否或反而限制译文灵活度→删除」**

### 4. 网文/现代文学语境优先
- 译名需符合**现代中文网络文学**使用习惯
- 避免过于古雅或学术化的译法
- 网络流行词、ID、外号保留原文或音译

### 4.5. 非英语原文禁止直接译为英语（强制规则）

**核心原则**：原文若不是英语，**不可**将 dst 直接写为英文单词或英文短语。必须翻译为中文。

**典型违规案例**（韩语外来语被错误地"还原"为英文）：
- ❌ `퍼시픽디벨롭` → Pacific Develop（应译中文：`太平洋开发` 或音译 `帕西菲克迪贝洛普`）
- ❌ `프라임에샛` → Prime Asset（应译中文：`优质资产` 或音译）
- ❌ `필로소피` → Philosophy（应译中文：`哲学` 或音译 `菲洛索菲`）
- ❌ `스테이크하우스` → Steakhouse（应译中文：`牛排馆`）

**判定流程**：
1. 检查 dst 是否为纯英文（无中文字符）
2. 若是：核查 src 是否为英文原文
   - src 是英文（如 `JN`、`Redfin`、`KNY`、`Phillip` 等英文缩写/英文人名）→ 保留 dst 英文 ✓
   - src 是韩文/日文外来语（如 `퍼시픽디벨롭`）→ **必须改为中文译名**
3. 若 src 是非英语外来语但确有保留英文必要（如商业品牌官方名）：
   - **必须先在审查报告中标注、给出推荐译法、等用户确认**后才能保留英文
   - 不允许 AI 自行决定保留英文

**例外（可保留英文）**：
- src 本身就是**英文字母 / 英文缩写 / 公司商号**：`JN` / `LA` / `KNY` / `Redfin` 等 — dst 自然保留
- src 是**英文特定场所名**（酒吧/夜店/俱乐部/餐厅/品牌店铺等）：可保留英文，如 `The Blue Note` / `Sky Lounge` 等
- 经用户明确确认的品牌/术语保留英文

**强制中文音译（不可保留英文）**：
- **英文人名**（含 src 是英文字母拼写的人名）必须按通用音译规则译为中文：
  - ✅ `Philip` / `Phillip` / `필립` → **菲利普**（不可保留 `Philip`）
  - ✅ `John` → 约翰 / `Mary` → 玛丽 / `David` → 大卫 / `Michael` → 迈克尔
  - ✅ `Spencer` → 斯宾塞 / `Hailey` → 海莉
  - **理由**：人名是角色术语的核心，必须有稳定中文译名以保证全文一致；保留英文会导致译文夹杂、风格不统一
- 同样适用于英文姓氏、英文全名（First Last 形式）

**判定优先级**：
1. src 是英文人名 → 通用中文音译
2. src 是英文场所/品牌/缩写 → 可保留英文
3. src 是非英语外来语但 dst 为英文 → 必须改为中文（推荐译法 + 用户确认）

### 5. 分类标准化（大类/子类格式）
- `角色` → 男性角色 / 女性角色 / 动物与非人角色 / 历史与知名人物 / 群体代称 / 称呼与头衔 / ID与外号
- `地点` → 特定现实地名 / 通用场景地名 / 奇幻与科幻地点 / 命名地理
- `组织机构` → 现代社会机构 / 奇幻与科幻阵营 / 派系与民间团体
- `小说设定` → 核心背景设定 / 网文机制元素 / 阶级与社会地位
- `能力技能` → 战斗与魔法技能 / 系统与网文异能 / 生活与职业技能
- `物品` → 现代日常物品 / 奇幻与科幻道具 / 设定专属特殊物
- 不属于任何子类时，自定义贴切子类，保持"大类/子类"格式

## §韩语专属规则块（--lang ko）

### 韩→中翻译要点
- **韩文人名结构**：姓（1字） + 名（1-2字），如 `이해든 (李海灯)`、`현재웅 (玄在雄)`
- **省略形式**：常去姓只称名（`해든`、`재웅`），译时去姓只称名
- **称谓后缀**（识别但译时多省略或意译）：
  - `씨` → 先生/小姐
  - `님` → 大人/先生
  - `오빠` → 哥哥（女对男）
  - `형` → 哥（男对男）
  - `누나` → 姐姐（男对女）
  - `언니` → 姐姐（女对女）
  - `선배` → 前辈
  - `선생님` → 老师
- **外来语转写**：카타카나-style 한글（`아이튜브`/`MCE 뮤직`），多源自英语，回译时保留英文或常见中文译名
- **网络用语来源**：DC Inside、더쿠、트위터、인스타、나무위키
- **网文流行**：romance fantasy（로판）、BL（브로맨스/BL）、转生（빙의/회귀物）

## §日语专属规则块（--lang ja）

### 日→中翻译要点
- **日文人名结构**：姓（汉字 1-2 字） + 名（汉字/假名 1-3 字），如 `佐藤太郎`
- **称谓后缀**（关键，日语此类极多）：
  - `さん` → 先生/小姐 / 省略
  - `くん` → 君（多用于男性同辈/晚辈）
  - `ちゃん` → 酱/小（昵称）
  - `様(さま)` → 大人/殿下
  - `先輩(せんぱい)` → 前辈
  - `後輩(こうはい)` → 后辈
  - `先生` → 老师/医生
  - `殿(どの)` → 阁下（古风）
  - `氏(し)` → 氏（正式书面）
  - **审查时**：保留称谓的术语条目应在 `info` 中标注 `称呼与头衔`，且与无后缀本名条目视为同一术语组
- **外来语**（カタカナ）：多为英文音译，常见和制英语。回译时优先用通行中文译名，无则保留英文或音译
- **拟声拟态语**（擬音語/擬態語）：日语此类极多（`ドキドキ` / `ふわふわ` / `しっとり`），多为副词/形容词性，**Tier C 时建议删除**
- **敬语层级**：タメ口（无敬语）/ 丁寧語（です/ます）/ 尊敬語 / 謙譲語 → 影响译文语气，但术语层面不直译敬语形态
- **汉字异读**：同一汉字不同假名读音（如 `中田`：なかた / なかだ）→ 视为不同术语
- **罗马字 ID**：保留原形（如 pixiv ID）
- **网络用语来源**：5ch（旧 2ch）、Twitter、pixiv、なろう、カクヨム
- **网文流行**：异世界転生 / 悪役令嬢 / 追放 / なろう系 / 乙女ゲー憑依

### 日语审查 Prompt（注入到每个术语的判定中）
```
你是顶级日中翻译复审专家。
绝对忠于「小说背景设定」与「术语所在原文参考」。
专有名词（人名/地名/组织/技能名）的一致性优先级高于字面准确性。
识别「同一角色的不同指代形式」（姓+名 / 仅姓 / 仅名 / +さん/くん/ちゃん/様 等），中文译名保持同源。
日语称谓后缀按上下文意译或省略：さん多省略、ちゃん译"小X"、様译"大人/殿下"、先輩译"前辈"。
擬音語/擬態語（如 ドキドキ/ふわふわ）若 Tier C 应大胆删除。
カタカナ外来语优先用通行中文译名，无则保留英文或音译。
同汉字异读视为不同术语。
译名风格符合现代中文网文/轻小说习惯（异世界転生、悪役令嬢、なろう系等）。
```

## 注意事项与边界

- **不生成** `modified.xlsx` / `modified.json` 变更日志（用户保留 `output.xlsx` 源文件可追溯）
- **不做多轮审查**（单轮审查 + 用户确认即终止；若用户要求再过一轮，重新调用 skill）
- **网络搜索失败**：记录在 `justification` 中"未找到相关搜索结果"，仍按上下文 + 规则判定
- **大型术语表**（>200 条）：分组处理时给出阶段性进度提示（"已完成 角色组 45/45，进入 地点组..."）
- **xlsx 文件被占用**（用户在 Excel 中打开）：写回失败时提示用户关闭后重试，**不创建带后缀的新文件**
